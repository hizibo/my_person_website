## 一、项目背景与目标

### 1.1 项目背景

当前个人/团队缺少统一的测试管理平台，自动化测试能力分散在各个独立项目中，缺乏统一入口和可视化管控。随着项目规模扩大，急需一个支持多种测试类型（接口、UI、App、性能）的综合测试平台。

### 1.2 总体目标

搭建一个**引擎化、高扩展**的一站式测试平台，以**测试执行引擎**为核心，支持多前端（Web/管理平台）接入，为测试团队提供从用例编写到报告输出的全链路能力。

### 1.3 远景定位（引擎式架构）

> **核心思路**：测试执行做成独立引擎，V1.0 搭框架，V2.0 开放接口直接对接前端，V3.0 接入专业用例管理平台（如 metersphere）**管控**引擎。

| 版本 | 阶段目标 | 架构特点 |
|------|---------|---------|
| **1.0（基础版）** | 搭框架、建能力 | 引擎核心框架 + pytest/Playwright/Locust 基座 |
| **2.0（开放版）** | 引擎接口化 | 引擎开放 REST/WebSocket 接口，**前端直接调用**，绕过 Java 后端 |
| **3.0（平台版）** | 接入用例管理平台 | metersphere / testPlatform 等 Python 用例平台通过接口管控引擎 |

**三者关系**：
```
V1.0: 前端工具页 ──→ Java后端 ──→ Python引擎
V2.0: 前端工具页 ──────────→ Python引擎（直接调用）
V3.0: Python用例管理平台 ───→ Python引擎
```

## 二、技术选型

### 2.1 测试框架

| 测试类型 | 框架选型 | 说明 |
|---------|---------|------|
| 接口自动化 | **pytest + requests** | Python 生态最成熟，社区活跃 |
| UI 自动化 | **Playwright** | 支持多浏览器（Chromium/Firefox/WebKit），API 友好 |
| App 自动化 | ~~pytest + Appium~~ | ~~复用 pytest 生态，移动端全覆盖~~ | ❌ V1.0 占位，后续扩展 |
| 性能测试 | **Locust** | Python 原生，支持分布式，协议丰富 |
| 测试报告 | **Allure** | 业界标准，可视化美观的报告 |

### 2.2 技术栈

| 层级 | 技术选型 |
|------|---------|
| **测试执行引擎** | pytest + requests + Playwright + Locust（核心引擎） |

| 数据库 | MySQL 8.0（管理数据存储） |
| 存储 | 测试脚本 Git 仓库 + Allure 报告存储 |

### 2.4 引擎核心接口（V2.0 开放）

> 以下是引擎 V2.0 计划开放的接口，**前端工具页直接调用**，无需经过 Java 后端。

| 接口 | 方法 | 说明 |
|------|------|------|
| 执行测试 | `POST /api/engine/execute` | 触发指定用例/套件执行 |
| 停止执行 | `POST /api/engine/stop/:task_id` | 停止正在执行的测试 |
| 查询状态 | `GET /api/engine/status/:task_id` | 查询执行状态 |
| 获取日志 | `GET /api/engine/logs/:task_id` | 获取实时日志流 |
| 查询报告 | `GET /api/engine/report/:report_id` | 获取报告摘要 |
| 下载报告 | `GET /api/engine/report/:report_id/download` | 下载 Allure 报告 ZIP |
| WebSocket 实时推送 | `WS /ws/engine/:task_id` | 推送执行进度、日志、状态 |
| 引擎健康检查 | `GET /api/engine/health` | 引擎健康状态 |
| 获取用例列表 | `GET /api/engine/cases` | 获取可用测试用例 |
| 获取环境配置 | `GET /api/engine/envs` | 获取可用测试环境列表 |

### 3.2 V2.0 功能模块（开放版）

> **核心变化**：引擎开放 REST/WebSocket 接口，前端**直接调用**，Java 后端退出执行路径（仅保留数据管理）。

#### 3.2.1 引擎接口开放

| 功能 | 描述 |
|------|------|
| REST API | 标准化接口，用例执行/停止/查询全部走引擎 |
| WebSocket 实时推送 | 执行日志、进度、状态实时推送到前端 |
| 认证鉴权 | API Key + JWT 双认证，接口安全可控 |
| 并发控制 | 限制同时执行的测试任务数（默认 3 个），防止资源耗尽 |
| 执行队列 | 任务排队机制，支持优先级（P0 可抢占），队列容量上限 100，超时 1h 自动终止 |
| 灰度环境 | 支持多套测试环境切换 |

#### 3.2.2 Swagger / OpenAPI 接口识别

> **核心功能**：导入 Swagger JSON/YAML 文档，自动识别所有接口，自动生成测试用例骨架，大幅降低接口测试的上手成本。

| 功能 | 描述 |
|------|------|
| Swagger 导入 | 支持导入 Swagger 2.0 / OpenAPI 3.0 JSON 或 YAML 文档 |
| 接口解析 | 自动解析所有路径（path）、方法（GET/POST/PUT/DELETE）、参数、请求体、响应码 |
| 用例生成 | 根据接口定义自动生成 pytest 用例骨架（含参数化） |
| 参数映射 | 自动映射 path param / query param / header / body |
| 响应校验 | 自动生成响应字段断言（type、required、enum） |
| 批量导入 | 支持指定 URL（`/swagger.json`）或上传文件 |
| 用例编辑 | 生成后可二次编辑，补充业务逻辑 |

> ⚠️ **技术注意**：OpenAPI 3.0 规范支持 `$ref` 循环引用、`oneOf`/`anyOf` 组合、嵌套对象数组、多级 schema 继承等高级特性。V2.0 `swagger_parser.py` 至少需支持 `$ref` 解析和常见 schema 组合关键字，仅解析平铺接口定义无法覆盖真实 Swagger 文档。

**使用流程**：

```
1. 输入 Swagger URL 或上传 Swagger 文件
         ↓
2. 解析接口，生成接口列表
         ↓
3. 选择要生成用例的接口
         ↓
4. 一键生成 pytest 用例骨架
         ↓
5. 补充参数值、断言逻辑
         ↓
6. 保存到 api/test_cases/
```

**Swagger 解析示例**：

```python
# 输入 Swagger JSON
swagger_doc = {
    "/api/note/add": {
        "post": {
            "tags": ["笔记"],
            "summary": "创建笔记",
            "parameters": [
                {"name": "title", "in": "body", "required": True},
                {"name": "content", "in": "body"},
                {"name": "categoryId", "in": "body"}
            ],
            "responses": {"200": {"description": "成功"}}
        }
    }
}

# 输出：自动生成以下用例骨架
def test_note_add():
    """POST /api/note/add - 创建笔记"""
    resp = client.post("/api/note/add", data={
        "title": "<TODO: 填写>",
        "content": "<TODO: 填写>",
        "categoryId": 1
    })
    assert resp["code"] == 200
```

## 七、风险评估

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| App 自动化环境复杂 | Android SDK 需要 2-3GB+，太重 | V1.0 占位，后续按需启用 |
| UI 自动化稳定性 | 元素定位容易受前端变更影响 | 采用稳定属性（data-testid）+ 兜底策略 |
| 性能测试资源 | 压测占用大量服务器资源 | 独立 Docker 容器执行，配额限制 |
| Python 服务运维 | 复用 python-tools 宿主机执行，无额外运维负担 | 与现有 python-tools 集成 |
| 测试脚本版本管理 | Git 分支与测试环境匹配问题 | 强制执行前拉取最新代码 |
| Python 依赖冲突 | test-engine 与现有 python-tools 共享环境，可能版本冲突 | 使用独立虚拟环境或 Docker 隔离 |
| Allure 报告存储膨胀 | 每次执行生成完整 HTML 报告，磁盘快速增长 | 仅保留最近 N 次报告 + 定时清理脚本 |
| Playwright 浏览器版本 | Playwright 强绑定浏览器版本，升级需同步 | 锁定 playwright 版本，升级走变更流程 |
| Locust 压测影响宿主机 | 压测占用大量端口/CPU，影响同机服务 | 独立容器 + CPU 限制 + 端口范围隔离 |
| Swagger 解析复杂度 | OpenAPI 3.0 的 $ref/oneOf/anyOf 等高级特性解析困难 | V2.0 优先支持常见模式，渐进增强 |

## 八、评审结论

> 以下内容已根据用户反馈确认（2026-05-17）：

| # | 问题 | 用户决策 | 说明 |
|---|------|---------|------|
| 1 | **App 自动化** | ❌ 暂不做 | Android SDK 需要 2-3GB+，太重；先专注接口+UI+性能 |
| 2 | **测试脚本存储** | ✅ 同项目 | 放 python-services/，作为工具集成到网站 |

| 4 | **CI/CD 集成** | ✅ 复用现有 | Jenkinsfile 已支持 python-services/ 变更检测，无需调整 |
| 5 | **报告查看** | ✅ 内嵌 | Allure 报告用 iframe 内嵌，技术成本与跳转相近 |
| **6** | **架构模式** | ✅ **引擎式** | 测试执行做成独立引擎，V1.0 框架 + V2.0 接口开放 + V3.0 对接用例管理平台 |
| **7** | **V1.0 定位** | ✅ **本地调试** | V1.0 不接前端/后端，纯本地可运行，预置网站用例 |

### 8.1 评审补充说明

- **V1.0 定位调整**：V1.0 纯本地调试，不依赖任何前端/后端，直接 `pytest` 命令运行，预置 15+ 网站用例，上线即能用
- **V2.0 Swagger 导入**：接口测试支持导入 Swagger JSON/YAML，自动解析接口并生成用例骨架，大幅降低接口测试门槛
- **App 自动化结论**：V1.0 不做 App 自动化，后续如需可单独扩展
- **CI/CD 零改动**：Jenkinsfile 的 `python-services/` 变更检测 + 独立重建机制天然覆盖测试平台
- **报告内嵌实现**：V2.0 才做前端，V2.0 前端通过 nginx 代理 Allure 报告静态路径 + `<iframe>` 嵌入工具页
- **引擎架构**：V1.0 本地 pytest → V2.0 FastAPI 开放接口 → V3.0 对接 metersphere 等用例管理平台

### 9.1 项目整体架构

#### 9.1.1 为什么这样分层？

```
前端工具页 ──HTTP──> Java 后端 ──HTTP──> Python 测试引擎
                    │                    │
                    │                    └──> 读写文件/执行命令
                    │
                    └──> MySQL（管理数据）
```

**为什么要用 Python 执行测试，而不是直接在 Java 里写？**

| 对比项 | 纯 Java 方案 | Python 方案（本项目） |
|--------|-------------|----------------------|
| 接口测试 | 需要额外学 HTTP Client 库 | requests 全球最成熟 |
| UI 测试 | Selenium Java 版 | Playwright Python 是官方主力 |
| 性能测试 | JMeter（XML配置） | Locust（Python 脚本，超灵活）|
| 学习曲线 | Java 测试框架较重 | Python 测试生态更轻量 |

**简单理解**：Python 是测试领域的"万能胶"，什么框架都能接。

#### 9.1.2 引擎架构演进（三个版本的调用关系）

**V1.0（Java 转发）**：
```
前端工具页 ──→ Java 后端 ──转发──→ Python 引擎
                              ↑
                         V1.0 接口（内部）
```
> V1.0 引擎只搭框架，接口由 Java 后端转发调用，作为过渡阶段。

**V2.0（引擎直连前端）**：
```
前端工具页 ──直接HTTP/WebSocket──→ Python 引擎
       ↑
   V2.0 开放接口

Java 后端（仅管理数据，非执行路径）
```
> V2.0 引擎开放标准化接口，前端直接调用，Java 后端退出执行路径。

**V3.0（用例管理平台管控）**：
```
Python 用例管理平台
  （metersphere 等）
        │
        ├── POST /api/engine/execute
        ├── WS /ws/engine/:id
        └── GET /api/engine/report/:id
        │
        ▼
   Python 引擎
   （被管控的执行单元）
```
> V3.0 引擎作为被管控的底层执行器，用例管理平台通过接口下发任务、获取结果。

**V1.0 本地调试 vs V2.0 引擎直连**

| 对比项 | 本地调试（V1.0） | 引擎直连（V2.0） |
|--------|----------------|-----------------|
| 部署方式 | 直接在宿主机运行 pytest | 引擎部署为服务，API 开放 |
| 前端交互 | 无，纯命令行本地调试 | 前端直接调用引擎接口 |
| 用例平台对接 | 不支持 | 用例平台直接管控引擎 |
| 实时性 | 无 | WebSocket 实时推送 |

### 9.2 common/ 公用模块详解

#### 9.2.1 http_client.py —— 为什么要封装 HTTP 请求？

**原始方式（不封装）**：
```python
# 每次请求都要写这么长...
response = requests.post(
    url="http://example.com/api/login",
    headers={"Content-Type": "application/json", "Authorization": token},
    json={"username": "admin", "password": "123456"},
    timeout=30
)
# 然后还要判断状态码...
if response.status_code == 200:
    data = response.json()
```

**封装后（简单优雅）**：
```python
# 初始化一次，后续直接用
client = HttpClient(base_url="http://example.com")
client.set_token(token)  # 设置一次 Token，后续自动带上

# 请求就这么简单
result = client.post("/api/login", data={"username": "admin", "password": "123456"})
```

**封装做了什么？**
| 功能 | 说明 |
|------|------|
| 自动带 Token | 登录后设置一次，后续所有请求自动带上 |
| 自动解析 JSON | 返回 Python 字典，不用每次 `.json()` |
| 统一错误处理 | HTTP 错误码自动抛异常，不用每次判断 |
| 请求日志 | 每次请求自动打印入参和出参，方便调试 |
| 超时控制 | 默认 30 秒超时，防止请求卡死 |

**维护指南**：
- 如果接口需要特殊 Header，在 `http_client.py` 里加一个 `set_header()` 方法
- 如果接口返回非 JSON，在 `request()` 方法里加一个参数控制解析方式

#### 9.2.2 assertions.py —— 什么是断言？为什么重要？

**断言 = 检查结果是否符合预期**

```python
# 不使用断言（很难发现问题）
response = client.get("/api/user/1")
user = response["data"]
print(user["name"])  # 只打印，不知道对不对

# 使用断言（错了立刻知道）
response = client.get("/api/user/1")
assert_response(response)
assert_equal(response["data"]["name"], "张三", "用户名应该是张三")
assert_contains(response["data"]["email"], "@example.com", "邮箱格式不对")
assert_code(response, 200)
```

**常用断言方法**：

| 方法 | 用途 | 示例 |
|------|------|------|
| `assert_equal(a, b)` | 相等 | 断言返回码是 200 |
| `assert_not_equal(a, b)` | 不等 | 断言用户 ID 不是 0 |
| `assert_contains(haystack, needle)` | 包含 | 断言响应包含某字符串 |
| `assert_not_contains(haystack, needle)` | 不包含 | 断言错误信息不包含敏感词 |
| `assert_true(value)` | 为真 | 断言 `is_active` 是 True |
| `assert_false(value)` | 为假 | 断言 `is_deleted` 是 False |
| `assert_length(array, length)` | 长度 | 断言返回 10 条数据 |
| `assert_regex(text, pattern)` | 正则 | 断言邮箱格式正确 |

**维护指南**：
- 如果需要特殊断言（如 JSON Schema 校验），在 `assertions.py` 加新方法
- 断言失败会自动截图/记录日志，方便定位问题

#### 9.2.3 logger.py —— 日志有什么用？

**日志 = 程序的"黑匣子"**

```python
import logger

log = logger.get_logger("登录测试")

log.info("开始执行登录测试")
log.debug("请求参数: {}".format(data))  # 调试信息，上线后不打印
log.info("收到响应: {}".format(response))
log.warning("响应时间较长: {}ms".format(duration))  # 警告，不影响流程
log.error("登录失败: {}".format(error))  # 错误，需要关注
```

**日志级别**：

| 级别 | 什么时候用 | 生产环境 |
|------|-----------|---------|
| DEBUG | 调试时看详细信息 | ❌ 不打印 |
| INFO | 正常流程记录 | ✅ 打印 |
| WARNING | 异常但不影响流程 | ✅ 打印 |
| ERROR | 错误，需要处理 | ✅ 打印 |

**日志输出位置**：
- 控制台（屏幕）
- 文件 `logs/test-2026-05-17.log`
- Allure 报告（嵌入 HTML）

**维护指南**：
- 日志文件过大？修改 `logger.py` 的 `maxBytes` 和 `backupCount`
- 需要输出到远程？加一个 `LoggerstashHandler` 即可

### 9.4 ui/ UI 自动化详解

#### 9.4.1 Page Object 模式（为什么需要？）

**不用 Page Object（反面教材）**：
```python
def test_login():
    browser.get("http://example.com/login")
    browser.find_element("xpath", "//input[@name='username']").send_keys("admin")
    browser.find_element("xpath", "//input[@name='password']").send_keys("123456")
    browser.find_element("xpath", "//button[@type='submit']").click()
    # 如果页面改版，input 的 name 变了...要改 100 个测试用例！
```

**用 Page Object（推荐）**：
```python
# 文件: ui/pages/login_page.py
class LoginPage:
    def __init__(self, browser):
        self.browser = browser
    
    def input_username(self, username):
        self.browser.find_element("xpath", "//input[@name='username']").send_keys(username)
    
    def input_password(self, password):
        self.browser.find_element("xpath", "//input[@name='password']").send_keys(password)
    
    def click_login(self):
        self.browser.find_element("xpath", "//button[@type='submit']").click()
    
    def login(self, username, password):
        self.input_username(username)
        self.input_password(password)
        self.click_login()


# 文件: ui/test_cases/test_login.py
def test_login():
    page = LoginPage(browser)
    page.login("admin", "123456")  # 页面改版？只改 LoginPage 一个文件！
    
    # 断言
    assert "欢迎" in browser.title
```

#### 9.4.2 常用元素定位方式

| 定位方式 | 示例 | 什么时候用 |
|---------|------|-----------|
| XPath | `//button[@id='submit']` | 通用，灵活 |
| CSS | `#submit` 或 `.btn` | 简单元素 |
| Text | `text=登录` | 按钮、链接 |
| ID | `id=username` | 最稳定，推荐 |
| Name | `name=password` | 表单字段 |

**推荐做法**：让前端开发给关键元素加 `data-testid` 属性：
```html
<button data-testid="login-btn">登录</button>
```

然后定位：`browser.find_element("data-testid", "login-btn")`

#### 9.4.3 智能等待（避免测试不稳定）

**问题**：元素还没加载完就去找，会报错！

**解决方案**：
```python
# ❌ 硬等（不稳定）
import time
time.sleep(3)  # 不管元素有没有，都等 3 秒

# ✅ 智能等（稳定）
browser.wait_for_element("data-testid", "submit-btn", timeout=10)
# 等元素出现，最多等 10 秒，超时自动报错
```

### 9.5 perf/ 性能测试详解

#### 9.5.1 Locust vs JMeter，我选哪个？

| 对比项 | JMeter | Locust（本项目） |
|--------|--------|-----------------|
| 脚本语言 | XML（复杂） | Python（简单） |
| 团队协作 | Git 合并 XML 很难 | 普通代码，Git 友好 |
| 分布式 | 配置复杂 | 命令行一行搞定 |
| 协议支持 | 全，但配置繁琐 | 主流够用 |
| 调试 | 需要 GUI | 直接 Python 调试 |

**结论**：团队小、用 Python、追求灵活性 → Locust 完胜！

#### 9.5.2 Locust 脚本示例

```python
# 文件: perf/locustfile.py
from locust import HttpUser, task, between
import random

class WebsiteUser(HttpUser):
    """模拟网站用户行为"""
    
    # 用户行为间隔 1-3 秒
    wait_time = between(1, 3)
    
    def on_start(self):
        """用户开始时执行（登录等）"""
        resp = self.client.post("/api/login", json={
            "username": "load_test",
            "password": "test123"
        })
        self.token = resp.json()["data"]["token"]
    
    @task(3)  # 权重 3，这个任务执行概率更高
    def browse_home(self):
        """浏览首页"""
        self.client.get("/")
    
    @task(1)
    def search_product(self):
        """搜索商品"""
        keywords = ["手机", "电脑", "平板", "耳机"]
        self.client.get(f"/api/search?q={random.choice(keywords)}")
    
    @task(1)
    def view_product(self):
        """查看商品详情"""
        product_id = random.randint(1, 1000)
        self.client.get(f"/api/product/{product_id}")
```

### 9.6 报告与集成详解

#### 9.6.1 Allure 报告是怎么工作的？

```
pytest 执行测试
     │
     ├── 每个用例执行结果写入 --> ~/.allure-results/
     │                              ├── 1-result.json
     │                              ├── 2-result.json
     │                              └── ...
     │
     └── allure generate 生成报告 --> report/
                                      ├── index.html
                                      ├── history/
                                      └── ...
```

**为什么选 Allure？**
- 测试报告好看（饼图、趋势图、时间线）
- 每个用例失败都能看到截图、日志、步骤
- 支持在 HTML 里嵌入附件（JSON、截图）

#### 9.6.2 怎么生成报告？

```bash
# 1. 运行测试（带上 --alluredir）
pytest api/test_cases/ -v --alluredir reports/allure-results

# 2. 生成 HTML 报告
allure generate reports/allure-results -o reports/html --clean

# 3. 本地查看
allure open reports/html

# 4. 服务端查看（前端内嵌）
# nginx 配置：/test-reports/ -> /opt/my-toolbox/python-services/test-platform/reports/html/
```

### 9.8 常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `ModuleNotFoundError: No module named 'pytest'` | 没装依赖 | `pip install -r requirements.txt` |
| `requests.exceptions.SSLError` | SSL 证书问题 | `client.verify_ssl = False` 或配置证书 |
| `Playwright: browser not installed` | 没装浏览器 | `playwright install chromium` |
| UI 测试经常超时 | 元素定位不稳定 | 用 `data-testid` 或增加等待时间 |
| 性能测试结果波动大 | 服务器资源不稳定 | 多跑几次取平均值 |
| Allure 报告乱码 | 编码问题 | 设置 `PYTHONIOENCODING=utf-8` |
| 报告没有截图 | 失败时没触发截图 | 检查 `conftest.py` 的 `pytest_runtest_makereport` |
